Shared codebase with web team. Expo modules for camera (property photos), push notifications, location.

Modular architecture with decorators. TypeScript end-to-end. Guards for multi-tenant auth. Easy microservice split.

Python (FastAPI)
ML scoring models in Python ecosystem. scikit-learn for collaborative filtering. Called async via message queue.
Primary Database

JSONB for flexible property attributes. Row-level security for tenant isolation. PostGIS for geo queries.

Geo-distance queries for location-based matching. Full-text on descriptions. Percolator queries for real-time requirement alerts.

03 Property ↔ Requirement Matching Engine

The matching engine is the core differentiator. It uses Elasticsearch's Percolator + a weighted ML scoring layer running asynchronously.

STEP 01
Requirement Posted
Buyer/broker creates requirement with params: type, location, budget, BHK, amenities, timeline
→
STEP 02
Percolator Index
Requirement stored as ES percolator query. New listings auto-trigger matching against all active requirements
→
STEP 03
Scoring Layer
Python worker scores matches 0–100 on: geo proximity, budget fit, config match, preference weights
→
STEP 04
Match Queue
Top N matches published to RabbitMQ. Deduplication prevents repeat notifications
→
STEP 05
Broker Alert
WhatsApp/push/email sent to broker with matched property card. CRM lead auto-created
→
STEP 06
Feedback Loop
Broker actions (accepted/rejected) feed back into ML model to improve future scoring

04 Multi-Tenant Data Isolation Strategy

We use a Shared Database, Separate Schema approach with row-level security — balancing cost efficiency with strict isolation.

SHARED LAYER
Platform Auth
Subscription/Billing
Broker Network (cross-tenant)
Global Matching Index
↕ tenant_id column on all shared data + PostgreSQL Row Level Security policies
ISOLATED LAYER
Brokerage A
CRM, Leads, Team
Brokerage B
CRM, Leads, Team
Brokerage C
CRM, Leads, Team
Brokerage N…
CRM, Leads, Team

JWT Tenant Scoping

Every JWT contains tenant_id. API gateway middleware injects tenant context into every DB query. No cross-tenant data leak possible at ORM level.

PostgreSQL RLS

Row-Level Security policies at database level as a second safety net. Even raw SQL queries cannot access another tenant's data without bypassing RLS.

Storage Isolation

S3 paths: /tenant-{id}/properties/... Signed URLs with tenant validation. Redis key prefixing: tenant:{id}:session:{token}.

07 Hardest Technical Challenges

01
Real-time Matching at Scale
When 10,000 active requirements must be checked against each new listing in near-real-time without overwhelming the system. Solution: ES Percolator inverts the problem — requirements wait as stored queries, listings flow through.
02
Multi-Tenant Data Isolation
Ensuring absolute data separation while sharing infrastructure. A bug in tenant scoping middleware could expose a brokerage's confidential pipeline to a competitor. RLS + middleware + automated isolation tests required.
03
Cross-Tenant Broker Network
The collaboration network crosses tenant boundaries. Defining exactly what data is shareable (property snapshot) vs. what stays isolated (client name, deal terms) requires careful permission modeling.
04
Duplicate Listing Detection
Multiple brokers often list the same property. Detecting duplicates across tenants using address normalization + embedding similarity without creating legal conflicts around ownership of the listing.
05
Matching Quality vs. Spam
A broker getting 50 low-quality matches per day will stop using the platform. Match quality scoring and daily digest limits + broker feedback loops are critical to product retention.
06
WhatsApp-to-CRM Pipeline
Brokers communicate on WhatsApp. Capturing unstructured WhatsApp conversations, extracting requirements via NLP and creating structured CRM records without workflow friction is a key UX challenge.

08 Why This Stack for Real Estate

PostgreSQL over MongoDB

Real estate data has complex relational integrity: property → ownership → documents → transactions. JSONB gives flexibility for property attributes while ACID transactions ensure deal data consistency. PostGIS enables native geo queries critical for location matching.

Elasticsearch for Matching

Percolator queries are the secret weapon — store requirements as queries, index properties as documents. This inverted approach means matching 10,000 requirements against a new listing is a single ES percolate call, not 10,000 SQL queries.

NestJS for Multi-Tenant API

NestJS guards and interceptors are perfect for injecting tenant context uniformly. Module architecture maps directly to business domains (listings, CRM, matching). TypeScript across stack reduces integration bugs between frontend and backend teams.

Kubernetes for Scale

Matching engine is the most compute-intensive component. K8s horizontal pod autoscaler lets the matching workers scale independently from API servers. During high activity (new project launches), matching capacity scales up automatically without affecting CRM performance.
